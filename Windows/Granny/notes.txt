What was fundamentally different

You moved from trying to execute commands via the ASP interpreter (WScript.Shell.Exec inside ASP) to actually running a native executable that you uploaded to disk (netcat). Those are different trust/permission paths:
WScript.Shell.Exec inside ASP is a COM call that can be blocked by server policy, script restrictions, or the IIS worker account. That produced the 80070005 Access is denied errors you saw.
Running a native binary you uploaded (via a batch or by ShellExecute/cmd) uses the OS process-launch path. In some IIS setups the worker account can spawn new processes if the call goes through a path the OS allows (for instance running a .bat via cmd), even when Exec/Exec-like COM calls are restricted. In short: native binary invocation can succeed where the COM Exec path fails.
You also used WebDAV to place the binary into a location the server can access. Even if the webserver refuses to serve .exe over HTTP, the file can still exist on disk and be invoked locally by the IIS process.
