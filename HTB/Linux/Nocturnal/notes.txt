"The backup command is run via proc_open and backgrounded with &.
The process is started by the web server (www-data), but it’s not attached to a real TTY or network socket.
When you try a direct reverse shell, the process may be sandboxed, lack network access, or be killed quickly.
The output you see on the webpage (attempted revshell execution) is from the command’s stdout/stderr, captured and displayed in the web page, not sent to your listener.
Why the writeup’s method works:

You first upload a shell script (using curl in the password field) so the file is present on disk.
Then you send a second request to execute that file, which is more reliable because it’s a simple local execution, not a complex network operation in a restricted context.
Direct reverse shell attempts often fail because:

The web server’s process context may block outbound connections.
The shell may not persist long enough to connect.
The command’s output is redirected to a log file, not to your listener.
Summary:

The two-step method (upload, then execute) works because it separates file transfer and execution, making it more reliable in restricted environments.
Direct reverse shell commands often fail due to web server restrictions, process context, or output handling."
