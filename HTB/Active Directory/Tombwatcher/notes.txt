# Technical Notes: Advanced ADCS Abuse, ESC15, and ESC3 Chaining (TombWatcher Endgame)
###################################################################################################

## ADCS, ESC15, and Certificate Request Agent Abuse

### Why This Works:
- **ADCS (Active Directory Certificate Services)** is a common escalation vector in modern AD environments. If a template is misconfigured (e.g., EnrolleeSuppliesSubject=TRUE, SchemaVersion=1), and the CA is unpatched (CVE-2024-49019), an attacker can inject arbitrary EKUs (application policies) into a certificate request. This is ESC15 ("EKUwu").
- **ESC15** allows you to request a certificate for any user (e.g., Administrator) and add the "Certificate Request Agent" EKU, even if the template does not natively allow it. This is possible because the CA does not properly restrict EKUs when EnrolleeSuppliesSubject is set and the template is v1.
- **Certificate Request Agent** is a special EKU that allows the holder to request certificates "on behalf of" other users (ESC3). This is a powerful privilege, as it enables you to impersonate any user for certificate-based authentication.

### Attack Chain Nuances:
1. **Request a Certificate with Certificate Request Agent EKU:**
   - You request a certificate for yourself (cert_admin) using the vulnerable template, injecting the Certificate Request Agent EKU. This is only possible due to ESC15.
   - The resulting certificate (agent.pfx) can now be used to request certificates for other users (ESC3 abuse).

2. **On-Behalf-Of Certificate Request (ESC3):**
   - Using the agent.pfx, you request a "User" template certificate for Administrator, specifying the on-behalf-of flag. This leverages the Certificate Request Agent privilege.
   - The CA issues a certificate for Administrator, which can be used for full authentication (Kerberos, LDAP, etc.).

3. **Authentication and Hash Extraction:**
   - With the Administrator certificate, you can use certipy-ad to obtain a TGT and extract the NT hash for Administrator, achieving full domain compromise.

### Key Technical Points:
- **Why not just request a "Client Authentication" certificate for Administrator?**
  - If the template does not have the Client Authentication EKU, the CA will not issue a certificate valid for Kerberos logon. ESC15 allows you to inject EKUs, but some CAs may still restrict usage. The Certificate Request Agent EKU is more reliably abusable for chaining to ESC3.
- **Why did some requests fail with CERT_E_WRONG_USAGE or rpc_s_access_denied?**
  - These errors occur if the certificate used for on-behalf-of does not have the correct EKU, or if the template/CA restricts who can perform these actions. The correct chain is: get a cert with Certificate Request Agent EKU (via ESC15), then use that for on-behalf-of requests.
- **Why is Schema Version 1 important?**
  - Only v1 templates are vulnerable to ESC15. Later schema versions restrict EKU injection.
- **Why does EnrolleeSuppliesSubject matter?**
  - This flag allows the requester to specify arbitrary subject/UPN/EKU fields, which is the root of the ESC15 vulnerability.
- **Why is the CA patch level critical?**
  - The November 2024 patch (CVE-2024-49019) fixes this by preventing arbitrary EKU injection. Unpatched CAs are vulnerable.

### Detection and Defense:
- Monitor for certificate requests with unusual EKUs or on-behalf-of usage.
- Restrict who can enroll in sensitive templates and who can act as Certificate Request Agent.
- Patch ADCS servers to prevent ESC15.
- Regularly audit template permissions and CA configuration.
